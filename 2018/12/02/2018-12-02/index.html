2018 AI开发者大会知识图谱专题总结

发表于 2018-12-02 | | 阅读次数:

2018年11月9日下午，参加了csdn举办的AI开发者大会的知识图谱分论坛，包括了下面五个口头报告。

漆桂林《知识图谱的表示和推理》 东南大学

上图是Yago的表示形式，三元组+标识，增强了图谱的表示能力。

几种知识的表示方式：

RDF->RDFS(subclass, domain, range)->OWL->KG Embedding(多种方式：如考虑邻居信息、考虑图的信息、考虑实体的文本描述信息等)

图谱的推理：从图谱中推断新知识
推理的任务：

  • 查询两个实体间的关系
  • 预测两个实体间的关系
  • 查询类别下的实体
  • 预测类别下的实体
  • 查询与某个实体有某种关系的其他实体
  • 检测图谱的不一致性
  • ……

推理的方法分类：

  • 逻辑推理（数据量大时，NC complexity）
  • 统计推理（PRA(s->t)，路径加权）
  • 传播扩散（spreading activation）

上图是图谱更新的框架，核心思路是两个：一、更新热点实体；二、更新与之相关更有可能变化的实体。

张扬《大规模通用知识图谱构建及应用》 百度

报告首先介绍了知识图谱在百度的应用案例，包括：搜索、推荐（可解释）等。

实体信息/服务聚合：以实体为粒度理解用户意图，聚合全网优质资源且展现更全面、更有价值的搜索结果，并提供畅游式浏览体验，让人和搜索引擎的交互更自然。
上图左边显示《复联3》上映时的搜索结果，右图显示《复联3》在电影下线后的搜索结果，左图关注电影订票等服务，右图关注电影的评论等信息。

上图是实体的卡片搜索，针对不同类型实体，定制模板展示。

基于知识图谱进行个性化推荐：

  • 百万关注点、千万关联关系->用户兴趣
  • 实现语义标签与热门实体的关联
  • 事件脉络？？？

基于知识图谱进行知识加工，写作。

BGraph是百度自研的图检索产品，支持十亿级节点，千亿级边。

具体到各个细分垂直领域：学术、教育、地理知识图谱。

图谱构建的模块都差不多，但知识补全与扩展这块需要重视，包括：关系挖掘、标签挖掘、评论挖掘、关注点挖掘、异构数据关联等。

PCNN(Piecewise Convolutional Neural Network) 参考刘康他们的论文Zeng2015
这里貌似提到他们的训练语料规模有百万级。

实现开放事实的不间断抽取，对于我们特定领域来说，关系和实体种类没那么多，相对没那么复杂。

我们的本体构建应按照自顶向下模式构建，以人工为主，强调准确性。

实体归一一般就是常说的实体对齐。

上图体现了主动收录知识的过程，已知“上市公司”的实体都有“股票代码”这个属性，但爱奇艺、苹果实体没有股票代码的属性，因此生成query，去做抽取、推理或人工补全的任务。

这里可以与我们阅读理解的思路对应起来理解，本质上阅读理解是适合做信息抽取的。

算法不能解决所有问题，要想实用化，一定要做好数据建设，一定要强调人机交互！！！

百度知识图谱主要处理数据集成在文本，后面要扩展到多媒体。

知识的表示要丰富化，能够表达流程、时序、函数、逻辑、模型等。

知识图谱要与用户和应用场景结合，服务垂直行业应用。

张伟 《电商领域知识图谱的构建与应用》 阿里巴巴

这个报告包括三部分：图谱在电商的应用、面临的问题和阿里的技术研究工作。

知识引擎——构建层：知识融合、知识建模、知识推理、知识获取。
图谱两种使用方式：一、显性卡片；二、隐性嵌入（KG embedding）

这里提到一个小创新，利用知识图谱来提高槽填充的准确性，利用KG去匹配实体，并排序。

图谱中的形式化知识需要总结，通过形式化知识，自动推理，形成标签。
我们图谱也有很多形式化知识，这些知识容易量化，在图谱更新时，进行提前推理形成标签。（这个应该是我们在图谱构建中需要重点建设的！）

推理引擎针对形式化知识，可解释性强，便于产品运营人员参与。
这里的treelink应该是阿里内部的叫法，实际是GBDT，参考https://blog.csdn.net/wanghai__/article/details/20217851

transE算法在大规模图谱上根本跑不起来，他们的经验是可以在ontology上使用，来补全关系。

报告没有细讲，值得仔细研究一下！

总结

这次会议提到知识图谱的构建和应用过程涉及：工具+算法+知识，工具包括推理工具、建模工具、可视化工具是可以对外分享公开的，但算法和知识不太适宜公开。

专家提到知识融合目前是图谱构建过程中的最大挑战，但需求分析更为关键，需要设计图谱的应用场景，才能指导图谱的构建工作。

下面是会上专家提到的几个关键点，很有感触，总结如下：

  1. 图谱存储：当前方案都是混合存储架构。阿里采用的是关系数据库宽表+在线图数据库，也有企业基于posgresql或者mongodb。在图谱规模不大，人力有限的情况下，neo4j二次开发是一个可行的思路，收费版本的价格也可接受。
  2. 知识推理：贯穿图谱构建的各个环节，包括查询重写、不一致检测、多跳查询等。
  3. 人机交互：算法解决不了知识图谱的构建，百度阿里也都构建了人机交互平台，改善图谱质量。
  4. 成本问题：做好需求分析，场景粒度需求，不一定要保证图谱100%的正确性，寻找质量和应用的平衡点.
